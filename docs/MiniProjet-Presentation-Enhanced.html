Phase 1 & 2 : Planification et POC
Phase 1 : Planification
  • Schema architectural dessine
  • Estimation des couts avec AWS Pricing Calculator

Phase 2 : Application de base
  • VPC (10.0.0.0/16) avec sous-reseaux publics et prives
  • EC2 t3.micro (Ubuntu) dans le sous-reseau public
  • Node.js 18 + Express + MySQL sur une seule instance
  • Application CRUD fonctionnelle via IP publique
Code Phase 2 (credentials en dur)
const db = mysql.createConnection({
  host: 'localhost',
  user: 'nodeapp',   password: 'student12',  // En dur !   database: 'STUDENTS'
});

Problemes :
- Mot de passe en clair dans le code
- Single point of failure (1 serveur)
- BDD sur la meme machine que l'app
Phase 3 : Decouplage des composants
Actions realisees
  • Ajout de sous-reseaux prives dans 2 AZs
  • Deploiement RDS MySQL dans subnet prive (DB Subnet Group)
  • Creation du secret "Mydbsecret" dans Secrets Manager
  • Security Group RDS : port 3306 uniquement depuis Web SG
Code Phase 3-4 (Secrets Manager)
const { SecretsManagerClient,
  GetSecretValueCommand }
  = require("@aws-sdk/client-secrets-manager");

async function getSecret() {
  const client = new SecretsManagerClient(
    { region: "us-east-1" });
  const response = await client.send(     new GetSecretValueCommand({       SecretId: "Mydbsecret"     }));   return JSON.parse(response.SecretString);
}
Zero credential en dur dans le code !
Phase 4 : Haute disponibilite et scalabilite
Application Load Balancer
  • Deploye sur 2 AZs (us-east-1a, 1b)
  • Health check sur '/' toutes les 30s
  • Point d'entree unique pour l'app
EC2 Auto Scaling
  • Launch Template avec AMI pre-configuree
  • ASG : min=2, max=4, desired=2
  • Target Tracking : CPU moyen 50%
Test de charge
  • Outil loadtest depuis Cloud9
  • Requetes massives sur URL de l'ALB
  • Observation du scaling automatique
