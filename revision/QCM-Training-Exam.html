PARTIE 1 — Cloud Fundamentals (Q1-10)

Q1. Quel modele de tarification est un avantage cle du cloud computing ?
  1. Investissement initial eleve (CapEx) avec couts mensuels faibles
  2. Tarification fixe annuelle quel que soit l'usage
  3. Payer uniquement ce que l'on consomme (pay-as-you-go)
  4. Tarification gratuite pour les 5 premieres annees
C — Le cloud remplace le CapEx (investissement) par de l'OpEx (depenses operationnelles a l'usage).
Q2. Quelle affirmation decrit le mieux le cloud computing ?
  1. Acheter des serveurs physiques et les installer dans son bureau
  2. Livraison de ressources informatiques a la demande via Internet
  3. Utiliser uniquement des logiciels open source
  4. Heberger des sites web exclusivement sur des serveurs locaux
B — Definition officielle du cloud computing selon le NIST et AWS.
Q3. Quel modele de service donne au client le PLUS de controle sur l'infrastructure ?
  1. SaaS
  2. PaaS
  3. IaaS
  4. FaaS
C — IaaS (ex: EC2) — Tu geres l'OS, les apps, le middleware. AWS gere uniquement le hardware et la virtualisation.
Q4. Amazon RDS est un exemple de quel modele de service ?
  1. IaaS
  2. PaaS
  3. SaaS
  4. DaaS
B — RDS est un service gere (AWS gere l'OS, le patching, les backups). Tu geres tes donnees et tes requetes = PaaS.
Q5. Combien d'Availability Zones (AZ) minimum par region AWS ?
  1. 1
  2. 2
  3. 4
  4. 6
B — Chaque region a au minimum 2 AZ (generalement 3+). Les AZ sont des datacenters physiquement separes.
Q6. Qu'est-ce qu'une Edge Location ?
  1. Un datacenter principal dans une region
  2. Un point de presence pour CloudFront et Route 53
  3. Un subnet dans un VPC
  4. Une zone de disponibilite secondaire
B — Les Edge Locations (400+) sont des points de cache pour le CDN CloudFront et le DNS Route 53.
Q7. Quel avantage du cloud permet de deployer en Asie, Europe et Amerique simultanement ?
  1. Economies d'echelle
  2. Depenses variables
  3. Deploiement mondial en minutes
  4. Fin de la maintenance datacenter
C — Avantage #6 : AWS a des regions partout dans le monde, tu peux deployer en quelques clics.
Q8. Quel modele de deploiement combine infrastructure sur site et cloud public ?
  1. Cloud public
  2. Cloud prive
  3. Cloud hybride
  4. Multi-cloud
C — Hybride = on-premises + cloud public. Utile quand certaines donnees doivent rester en local (compliance).
Q9. Quel est le service AWS qui permet de deployer une app web sans gerer l'infrastructure (PaaS) ?
  1. Amazon EC2
  2. AWS Lambda
  3. AWS Elastic Beanstalk
  4. Amazon S3
C — Elastic Beanstalk = PaaS. Tu upload ton code, AWS gere EC2, ALB, ASG, monitoring.
Q10. Les Availability Zones d'une meme region sont :
  1. Dans le meme batiment physique
  2. Physiquement separees mais connectees par reseau tres rapide
  3. Situees dans des pays differents
  4. Accessibles uniquement via VPN
B — Les AZ sont dans des batiments separes (alimentation, refroidissement independants) mais relies par fibre a faible latence (<1ms).

PARTIE 2 — VPC et Reseau (Q11-25)

Q11. Un VPC avec le CIDR 10.0.0.0/16 contient combien d'adresses IP ?
  1. 256
  2. 65 536
  3. 16 777 216
  4. 512
B — /16 = 2^(32-16) = 2^16 = 65 536 adresses IP.
Q12. Combien d'adresses IP AWS reserve-t-il par subnet ?
  1. 2
  2. 3
  3. 5
  4. 10
C — AWS reserve 5 IPs : adresse reseau, routeur VPC, DNS, usage futur, broadcast.
Q13. Qu'est-ce qui rend un subnet "public" ?
  1. Il a un nom qui contient "public"
  2. Sa route table a une route 0.0.0.0/0 vers un Internet Gateway
  3. Il est dans la premiere AZ de la region
  4. Il a un Security Group qui autorise tout le trafic
B — Un subnet est public quand sa table de routage envoie le trafic Internet (0.0.0.0/0) vers un IGW.
Q14. Combien d'Internet Gateway (IGW) peut-on attacher a un VPC ?
  1. Autant que de subnets
  2. 1 par AZ
  3. 1 seul par VPC
  4. Illimite
C — 1 IGW par VPC maximum. L'IGW est hautement disponible par defaut (gere par AWS).
Q15. Ou doit etre place un NAT Gateway ?
  1. Dans un subnet prive
  2. Dans un subnet public
  3. En dehors du VPC
  4. Dans l'Internet Gateway
B — Le NAT Gateway est dans un subnet public (il a besoin de l'IGW pour acceder a Internet). Les instances privees routent vers lui.
Q16. A quoi sert un NAT Gateway ?
  1. Permettre a Internet d'acceder aux instances privees
  2. Permettre aux instances privees de sortir vers Internet sans etre accessibles
  3. Connecter deux VPC entre eux
  4. Chiffrer le trafic entre subnets
B — NAT = sortie uniquement. Les instances privees peuvent telecharger des MAJ mais ne sont pas joignables depuis Internet.
Q17. Les Security Groups sont :
  1. Stateless — il faut configurer entree ET sortie
  2. Stateful — si l'entree est autorisee, la sortie l'est automatiquement
  3. Appliques au niveau du subnet
  4. Capables de regles DENY
B — SG = stateful. Si tu autorises le trafic entrant sur le port 80, la reponse sortante est automatiquement autorisee.
Q18. Les NACL sont :
  1. Stateful, au niveau de l'instance
  2. Stateless, au niveau du subnet
  3. Stateful, au niveau du subnet
  4. Stateless, au niveau de l'instance
B — NACL = stateless + niveau subnet. Il faut configurer les regles inbound ET outbound separement.
Q19. Par defaut, un Security Group :
  1. Autorise tout le trafic entrant et sortant
  2. Bloque tout le trafic entrant et sortant
  3. Bloque tout le trafic entrant, autorise tout le trafic sortant
  4. Autorise tout le trafic entrant, bloque tout le sortant
C — SG par defaut : tout BLOQUE en entrant (securite), tout AUTORISE en sortant.
Q20. Comment autoriser ton serveur web EC2 a se connecter a ta BDD RDS (port 3306) de maniere securisee ?
  1. Ouvrir le port 3306 au monde entier (0.0.0.0/0) dans le SG de RDS
  2. Mettre le SG du serveur web comme source dans le SG de RDS pour le port 3306
  3. Desactiver le SG de RDS
  4. Utiliser un NAT Gateway
B — Chainage de SG : le SG de RDS autorise le port 3306 depuis le SG de l'EC2 web. Jamais ouvrir 3306 a 0.0.0.0/0 !
Q21. Quelle est la principale difference entre SG et NACL pour les regles ?
  1. SG supporte ALLOW et DENY, NACL uniquement ALLOW
  2. SG uniquement ALLOW, NACL supporte ALLOW et DENY
  3. Les deux supportent ALLOW et DENY
  4. Aucun ne supporte DENY
B — SG = ALLOW uniquement. NACL = ALLOW + DENY (avec evaluation par numero de regle).
Q22. Un VPC A est peere avec B, et B est peere avec C. A peut-il communiquer avec C ?
  1. Oui, automatiquement
  2. Oui, apres mise a jour des route tables
  3. Non, le VPC peering est non transitif
  4. Oui, via le NAT Gateway de B
C — VPC Peering = non transitif. Pour que A parle a C, il faut un peering direct A ↔ C.
Q23. Un subnet peut-il s'etendre sur plusieurs AZ ?
  1. Oui, c'est le comportement par defaut
  2. Oui, si on le configure explicitement
  3. Non, un subnet est toujours dans UNE seule AZ
  4. Oui, mais uniquement dans les subnets publics
C — 1 subnet = 1 AZ. Pour la haute dispo, tu crees des subnets dans plusieurs AZ.
Q24. Quelle est la route table d'un subnet PRIVE typique ?
  1. 10.0.0.0/16 → local, 0.0.0.0/0 → igw-xxx
  2. 10.0.0.0/16 → local, 0.0.0.0/0 → nat-xxx
  3. 0.0.0.0/0 → local
  4. 10.0.0.0/16 → igw-xxx
B — Subnet prive : trafic interne via "local", trafic Internet via NAT Gateway (pas IGW).
Q25. Une Elastic IP est gratuite quand :
  1. Elle est toujours gratuite
  2. Elle est associee a une instance EC2 en cours d'execution
  3. Elle n'est associee a rien
  4. Elle est dans un subnet prive
B — EIP gratuite quand utilisee (attachee a une instance running). Payante si non utilisee (AWS evite le gaspillage d'IPs).

PARTIE 3 — EC2 (Q26-38)

Q26. Qu'est-ce qu'une AMI ?
  1. Un type d'instance EC2
  2. Une image/template d'un serveur (OS + config + logiciels)
  3. Un service de monitoring
  4. Un protocole reseau
B — AMI = Amazon Machine Image. C'est le "modele" pour lancer une instance (OS + logiciels pre-installes).
Q27. Quelle famille d'instance EC2 est optimisee pour le machine learning (GPU) ?
  1. T (General Purpose)
  2. C (Compute Optimized)
  3. P / G (Accelerated Computing)
  4. R (Memory Optimized)
C — Les familles P et G ont des GPU pour le ML, le rendu graphique, et le calcul parallele.
Q28. Le User Data d'une instance EC2 s'execute :
  1. A chaque redemarrage de l'instance
  2. Uniquement au premier lancement de l'instance
  3. Quand on le declenche manuellement
  4. Toutes les heures automatiquement
B — User Data = script execute une seule fois au premier boot (en tant que root). Utilise pour l'installation initiale.
Q29. Quel mode d'achat EC2 offre la plus grande reduction mais peut etre interrompu ?
  1. On-Demand
  2. Reserved Instances
  3. Spot Instances
  4. Dedicated Hosts
C — Spot = jusqu'a -90% mais AWS peut reprendre l'instance avec 2 min de preavis. Pour du calcul non critique.
Q30. Une application tourne 24/7 de maniere stable depuis 2 ans. Comment optimiser les couts ?
  1. Utiliser des Spot Instances
  2. Utiliser des Reserved Instances ou Savings Plans
  3. Utiliser du On-Demand
  4. Eteindre l'instance la nuit
B — Charge stable 24/7 = Reserved Instances ou Savings Plans (engagement 1-3 ans, reduction jusqu'a 72%).
Q31. Comment donner a une EC2 le droit d'acceder a S3 de maniere securisee ?
  1. Mettre les AWS Access Keys dans le code de l'application
  2. Stocker les Access Keys dans un fichier .env sur le serveur
  3. Attacher un IAM Role a l'instance EC2
  4. Ouvrir le port S3 dans le Security Group
C — TOUJOURS utiliser un IAM Role pour les services AWS. Jamais mettre des credentials dans le code ou des fichiers.
Q32. Quelle instance fait partie du AWS Free Tier ?
  1. t3.large
  2. m5.xlarge
  3. t2.micro
  4. c5.2xlarge
C — t2.micro = Free Tier (750 heures/mois pendant 12 mois).
Q33. Pour quelles licences logicielles utiliserait-on des Dedicated Hosts ?
  1. Open source (Linux, Apache)
  2. Licences liees au hardware physique (Oracle, Windows Server)
  3. SaaS (Gmail, Slack)
  4. Licences AWS natives
B — Dedicated Hosts = serveur physique dedie. Necessaire pour les licences logicielles qui se comptent par socket/core physique.
Q34. A quelle adresse une instance EC2 peut-elle acceder a ses propres metadonnees ?
  1. http://127.0.0.1/metadata
  2. http://169.254.169.254/latest/meta-data/
  3. http://aws.amazon.com/ec2/meta
  4. http://10.0.0.1/instance-info
B — 169.254.169.254 est l'adresse speciale pour les metadonnees EC2 (IP, role IAM, AZ, etc.).
Q35. Quel Placement Group minimise la latence entre instances ?
  1. Spread
  2. Partition
  3. Cluster
  4. Distributed
C — Cluster = toutes les instances dans le meme rack. Latence minimale. Pour le HPC.
Q36. Quelle famille d'instance pour une base de donnees en memoire qui a besoin de beaucoup de RAM ?
  1. C (Compute Optimized)
  2. T (General Purpose)
  3. R (Memory Optimized)
  4. I (Storage Optimized)
C — R = Memory Optimized. Pour les BDD in-memory, caches, analytics temps reel.
Q37. Que se passe-t-il quand une Spot Instance est interrompue par AWS ?
  1. AWS donne 30 minutes de preavis
  2. AWS donne 2 minutes de preavis
  3. L'instance est preservee indefiniment
  4. AWS ne peut pas interrompre une Spot Instance
B — Spot = 2 min de preavis avant interruption. L'appli doit etre tolerante aux interruptions.
Q38. Savings Plans vs Reserved Instances : quelle est la difference principale ?
  1. Savings Plans offrent plus de reduction
  2. Savings Plans sont plus flexibles (pas lies a un type d'instance specifique)
  3. Reserved Instances sont plus flexibles
  4. Il n'y a aucune difference
B — Savings Plans = engagement sur un montant $/h, mais tu peux changer de type d'instance, OS, region. Plus flexible que les RI.

PARTIE 4 — Stockage S3, EBS, EFS (Q39-48)

Q39. Quelle est la durabilite de S3 Standard ?
  1. 99.9%
  2. 99.99%
  3. 99.999999999% (11 neufs)
  4. 100%
C — S3 = 11 neufs de durabilite. Sur 10 millions d'objets, tu en perds 1 tous les 10 000 ans en moyenne.
Q40. Quel service pour heberger un site web statique (HTML/CSS/JS) sans serveur ?
  1. Amazon EC2
  2. Amazon RDS
  3. Amazon S3
  4. AWS Lambda
C — S3 Static Website Hosting = heberger un site statique sans gerer de serveur. Tres economique.
Q41. Quelle classe S3 pour archiver des donnees qu'on ne consulte quasi jamais (recuperation en 12h acceptable) ?
  1. S3 Standard
  2. S3 Standard-IA
  3. S3 Glacier Flexible Retrieval
  4. S3 Glacier Deep Archive
D — Deep Archive = le moins cher, recuperation en 12h. Pour l'archivage legal longue duree.
Q42. Comment reduire automatiquement les couts de stockage S3 au fil du temps ?
  1. Supprimer manuellement les anciens fichiers
  2. Configurer des S3 Lifecycle Rules
  3. Utiliser un cron job sur EC2
  4. Changer le type de bucket
B — Lifecycle Rules = transitions automatiques (Standard → IA → Glacier → Deep Archive) et expirations.
Q43. Un volume EBS peut etre attache a :
  1. Plusieurs instances EC2 dans la meme AZ
  2. Une seule instance EC2, dans la meme AZ
  3. Plusieurs instances EC2 dans differentes AZ
  4. N'importe quelle instance dans n'importe quelle region
B — EBS = 1 instance EC2 dans la meme AZ. Exception : io2 Multi-Attach (multi-instance mais meme AZ, cas rare).
Q44. Quel type de volume EBS pour une BDD haute performance (Oracle) ?
  1. gp3 (General Purpose SSD)
  2. io2 (Provisioned IOPS SSD)
  3. st1 (Throughput Optimized HDD)
  4. sc1 (Cold HDD)
B — io2 = jusqu'a 64 000 IOPS. Pour les BDD critiques qui ont besoin de performances I/O tres elevees.
Q45. Quel service de stockage est accessible par PLUSIEURS instances EC2 dans PLUSIEURS AZ ?
  1. EBS
  2. Instance Store
  3. EFS
  4. Aucun
C — EFS = systeme de fichiers partage NFS, accessible par N instances dans toutes les AZ d'une region.
Q46. Un EBS Snapshot est stocke dans :
  1. L'instance EC2
  2. Amazon S3 (gere par AWS)
  3. Un autre volume EBS
  4. Amazon Glacier
B — Les snapshots EBS sont stockes dans S3 (gere par AWS, tu ne les vois pas dans tes buckets). Peuvent etre copies cross-region.
Q47. Quelle est la taille maximale d'un objet dans S3 ?
  1. 5 Go
  2. 5 To
  3. 50 Go
  4. Illimitee
B — Un objet S3 peut faire jusqu'a 5 To. Au-dela de 5 Go, il faut utiliser le multipart upload.
Q48. Le Versioning S3 sert a :
  1. Accelerer l'acces aux fichiers
  2. Reduire les couts de stockage
  3. Conserver toutes les versions d'un objet et se proteger des suppressions accidentelles
  4. Chiffrer les fichiers
C — Versioning = chaque modification cree une nouvelle version. Meme un DELETE ne supprime pas vraiment (il ajoute un delete marker).

PARTIE 5 — RDS et Bases de donnees (Q49-55)

Q49. Multi-AZ RDS sert a :
  1. Ameliorer les performances de lecture
  2. Assurer la haute disponibilite (failover automatique)
  3. Reduire les couts
  4. Permettre les connexions depuis Internet
B — Multi-AZ = HAUTE DISPONIBILITE. Copie synchrone dans une autre AZ, failover automatique si la primaire tombe.
Q50. Read Replicas RDS servent a :
  1. Assurer la haute disponibilite
  2. Ameliorer les performances en LECTURE
  3. Chiffrer les donnees
  4. Sauvegarder la BDD
B — Read Replica = copie en lecture seule. On redirige les SELECT vers les replicas pour soulager la BDD principale.
Q51. Peut-on lire directement depuis le standby Multi-AZ ?
  1. Oui, comme un Read Replica
  2. Non, le standby est invisible et n'accepte aucune requete
  3. Oui, mais seulement en lecture
  4. Oui, apres configuration manuelle
B — Le standby Multi-AZ est en veille. Il ne recoit aucun trafic applicatif. Il prend le relais uniquement en cas de panne.
Q52. Amazon Aurora est compatible avec :
  1. MongoDB et Redis
  2. MySQL et PostgreSQL
  3. SQL Server et Oracle
  4. DynamoDB
B — Aurora = MySQL ou PostgreSQL compatible, mais 5x plus rapide que MySQL et 3x que PostgreSQL. Stocke 6 copies sur 3 AZ.
Q53. DynamoDB est :
  1. Une BDD relationnelle SQL
  2. Une BDD NoSQL cle-valeur serverless
  3. Un service de cache en memoire
  4. Un data warehouse
B — DynamoDB = NoSQL (cle-valeur + document), serverless, latence en millisecondes, scaling automatique.
Q54. Quel avantage RDS n'offre PAS par rapport a MySQL sur EC2 ?
  1. Backups automatiques
  2. Patching automatique de l'OS
  3. Acces SSH au serveur de BDD
  4. Multi-AZ en 1 clic
C — RDS = service gere, donc PAS d'acces SSH. C'est le compromis : moins de controle mais moins de maintenance.
Q55. DAX est :
  1. Un moteur de BDD relationnelle
  2. Un cache en memoire pour DynamoDB (latence microseconde)
  3. Un outil de backup pour RDS
  4. Un service de monitoring
B — DAX = DynamoDB Accelerator. Cache compatible DynamoDB, latence en microsecondes au lieu de millisecondes.

PARTIE 6 — IAM et Securite (Q56-65)

Q56. Le principe du moindre privilege signifie :
  1. Donner les permissions admin a tout le monde pour simplifier
  2. Ne donner que les permissions strictement necessaires
  3. Utiliser uniquement le compte root
  4. Desactiver toutes les permissions par defaut
B — Moindre privilege = chaque user/service n'a QUE les permissions dont il a besoin. Principe fondamental de securite AWS.
Q57. Quand utiliser un IAM Role au lieu d'un IAM User ?
  1. Pour une personne qui se connecte a la console AWS
  2. Pour un service AWS (EC2, Lambda) qui accede a d'autres services
  3. Pour stocker des mots de passe
  4. Pour creer un VPC
B — IAM Role = pour les services AWS. Credentials temporaires, pas de cles permanentes dans le code. IAM User = pour les humains.
Q58. Quel service enregistre CHAQUE appel API effectue sur un compte AWS ?
  1. CloudWatch
  2. CloudTrail
  3. AWS Config
  4. GuardDuty
B — CloudTrail = audit trail. Qui a fait quoi, quand, depuis ou. Chaque appel API est enregistre.
Q59. Ou stocker le mot de passe de la BDD dans une application AWS ?
  1. En dur dans le code source
  2. Dans un fichier .env sur le serveur
  3. Dans AWS Secrets Manager
  4. Dans un commentaire du code
C — Secrets Manager = stockage securise + rotation automatique des secrets. L'app les recupere via API au runtime.
Q60. Dans le modele de responsabilite partagee, QUI est responsable du patching de l'OS d'une instance EC2 ?
  1. AWS
  2. Le client
  3. Les deux
  4. Personne
B — EC2 = IaaS. Le client gere l'OS, le patching, la configuration. AWS gere le hardware physique.
Q61. Quel service protege contre les attaques DDoS ?
  1. AWS WAF
  2. AWS Shield
  3. AWS GuardDuty
  4. AWS Inspector
B — Shield Standard = protection DDoS gratuite et automatique. Shield Advanced = protection avancee ($3000/mois).
Q62. AWS WAF protege contre :
  1. Les attaques DDoS volumetriques
  2. Les attaques applicatives (SQL injection, XSS)
  3. Le vol de hardware dans les datacenters
  4. Les erreurs de configuration IAM
B — WAF = Web Application Firewall. Protege couche 7 : SQL injection, XSS, bots. Se place devant ALB/CloudFront/API GW.
Q63. Un bucket S3 est accidentellement rendu public. Dans le modele de responsabilite partagee, c'est la faute de :
  1. AWS
  2. Le client
  3. S3 est toujours public par defaut
  4. Personne, c'est normal
B — La configuration des ressources (SG, bucket policies, permissions) est la responsabilite du client.
Q64. Quel type de chiffrement protege les donnees stockees sur disque ?
  1. Chiffrement en transit
  2. Chiffrement au repos (at rest)
  3. Chiffrement SSL
  4. Chiffrement TLS
B — At rest = sur disque (EBS, S3, RDS). En transit = sur le reseau (HTTPS/TLS). Les deux sont recommandes.
Q65. MFA signifie :
  1. Multiple File Access
  2. Multi-Factor Authentication
  3. Managed Firewall Application
  4. Main Failover Architecture
B — MFA = mot de passe + deuxieme facteur (code du telephone). Obligatoire sur le compte root AWS.

PARTIE 7 — ALB, Auto Scaling, CloudFormation (Q66-72)

Q66. Quelle est la difference entre un ALB et un NLB ?
  1. ALB = couche 4 (TCP), NLB = couche 7 (HTTP)
  2. ALB = couche 7 (HTTP/HTTPS), NLB = couche 4 (TCP/UDP)
  3. ALB est gratuit, NLB est payant
  4. Il n'y a pas de difference
B — ALB = Application LB (couche 7, HTTP, routage par URL/host). NLB = Network LB (couche 4, TCP, ultra-rapide).
Q67. Le health check d'un ALB echoue pour une instance. Que se passe-t-il ?
  1. L'ALB continue d'envoyer du trafic normalement
  2. L'ALB arrete d'envoyer du trafic a cette instance
  3. L'ALB redemarr l'instance
  4. L'ALB envoie une alerte mais continue
B — L'instance est marquee "unhealthy" et retiree du target group. Plus aucun trafic ne lui est envoye.
Q68. Dans un ASG, la "Desired Capacity" est :
  1. Le nombre maximum d'instances
  2. Le nombre minimum d'instances
  3. Le nombre d'instances que l'ASG essaie de maintenir a l'instant T
  4. Le cout souhaite par mois
C — Desired = nombre cible actuel. L'ASG ajuste pour le maintenir (entre Min et Max).
Q69. Quel type de scaling policy : "maintenir le CPU moyen a 50%" ?
  1. Simple Scaling
  2. Step Scaling
  3. Target Tracking Scaling
  4. Scheduled Scaling
C — Target Tracking = l'ASG ajuste le nombre d'instances pour garder une metrique a une valeur cible.
Q70. Le scaling HORIZONTAL signifie :
  1. Augmenter la taille d'un serveur (plus de CPU/RAM)
  2. Ajouter plus de serveurs
  3. Deplacer le serveur dans une autre region
  4. Activer le chiffrement
B — Horizontal = ajouter des instances (scale out). Vertical = augmenter la taille (scale up). Horizontal est presque toujours la reponse en QCM.
Q71. Quelle section est OBLIGATOIRE dans un template CloudFormation ?
  1. Parameters
  2. Description
  3. Resources
  4. Outputs
C — Resources est la seule section obligatoire. C'est la liste des ressources AWS a creer.
Q72. Que se passe-t-il si la creation d'une stack CloudFormation echoue ?
  1. Les ressources deja creees restent en place
  2. CloudFormation fait un rollback automatique (tout est supprime)
  3. AWS envoie un email et attend
  4. Il faut supprimer manuellement les ressources
B — En cas d'echec, CloudFormation supprime automatiquement tout ce qui a ete cree (rollback). Tout ou rien.

PARTIE 8 — Services avances et scenarios (Q73-80)

Q73. Comment decoupler deux microservices pour ameliorer la resilience ?
  1. Les deployer sur la meme instance EC2
  2. Mettre une queue SQS entre les deux
  3. Utiliser un Security Group commun
  4. Les connecter directement via HTTP
B — SQS = file d'attente. Si un service tombe, les messages attendent. L'autre service les traite quand il est pret.
Q74. SNS vs SQS : quelle est la difference ?
  1. SNS est un stockage, SQS est un reseau
  2. SNS = 1 message vers N abonnes (pub/sub), SQS = file d'attente (1 pour 1)
  3. SQS = 1 message vers N abonnes, SNS = file d'attente
  4. Il n'y a pas de difference
B — SNS = pub/sub (haut-parleur). SQS = queue (comptoir). SNS+SQS = fan-out pattern (1 message → N queues).
Q75. Comment reduire la latence d'un site pour des utilisateurs en Asie (site heberge en Europe) ?

  2. Utiliser Amazon CloudFront (CDN)
  3. Ajouter un Read Replica RDS
  4. Utiliser un Security Group plus permissif
B — CloudFront met en cache le contenu dans les edge locations proches des utilisateurs. Reduit drastiquement la latence.

  4. Inclus dans le cout de l'instance
