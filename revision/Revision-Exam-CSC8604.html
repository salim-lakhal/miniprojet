1. Introduction au Cloud Computing

Definition

Le cloud computing = livraison de ressources informatiques (serveurs, stockage, bases de donnees, reseau) a la demande via Internet avec une tarification a l'usage (pay-as-you-go).

Les 6 avantages du cloud (a connaitre par coeur)

#AvantageExplication detaillee
1Depenses variablesAu lieu d'investir dans un datacenter (CapEx), tu paies uniquement ce que tu consommes (OpEx). Pas de cout initial.
2Economies d'echelleAWS achete des centaines de milliers de serveurs → cout unitaire tres bas → repercute sur les clients
3Plus besoin de deviner la capaciteOn-premise : tu achetes pour le pic → gaspillage. Cloud : tu scales exactement selon la demande
4Vitesse et agiliteLancer un serveur en 2 min au lieu d'attendre 2 semaines pour commander du hardware
5Fin de la maintenance datacenterPlus besoin de gerer le refroidissement, l'alimentation, les cables, la securite physique
6Deploiement mondial en minutesTu peux deployer ton app dans 30+ regions dans le monde en quelques clics

3 modeles de service (IaaS / PaaS / SaaS)

ModeleToi tu geresAWS gereExemples AWS
IaaSOS, apps, donnees, middlewareHardware, reseau, virtualisationEC2, VPC, EBS
PaaSTon code et tes donneesOS, serveur, scaling, patchingElastic Beanstalk, RDS
SaaSRien (tu utilises)Absolument toutGmail, Salesforce, Dropbox

Astuce QCM : Plus tu montes vers SaaS, moins tu as de controle mais moins tu as de travail.

3 modeles de deploiement

  • Cloud public — Tout chez un fournisseur (AWS, Azure, GCP). Le plus courant, le plus flexible
  • Cloud prive (on-premises) — Infrastructure dans ton propre datacenter avec des outils cloud (ex: VMware, OpenStack)
  • Cloud hybride — Mix des deux. Ex: donnees sensibles en prive, reste dans le cloud public. AWS propose AWS Outposts pour ca

Infrastructure mondiale AWS

Important : Chaque region a au minimum 2 AZ. Les AZ sont physiquement separees (different batiment, alimentation, reseau) mais connectees par fibre tres rapide (latence < 1ms). C'est ca qui permet le Multi-AZ.

QCM : Quel est l'avantage principal du cloud par rapport a un datacenter sur site ?
R : Remplacer les depenses initiales (CapEx) par des depenses variables (OpEx) a l'usage
QCM : Combien d'AZ minimum par region AWS ?
R : 2 (et generalement 3 ou plus)

3. VPC — Virtual Private Cloud (RESEAU) — EN DETAIL

C'est quoi un VPC ?

Un VPC est ton propre reseau prive virtuel dans AWS. Tu as le controle total sur :

  • La plage d'adresses IP (CIDR block)
  • La creation de subnets
  • La configuration des tables de routage
  • Les passerelles reseau (IGW, NAT)
  • Les regles de securite (SG, NACL)
VPC = ton immeuble prive | Subnets = les etages | IGW = la porte d'entree vers la rue | NAT = fenetre pour voir dehors sans etre vu | Route Table = panneaux directionnels | SG = vigile a chaque porte | NACL = vigile a l'entree de chaque etage

CIDR Block (les adresses IP)

Quand tu crees un VPC, tu choisis une plage d'adresses IP en notation CIDR.

CIDRNombre d'IPsUtilisation
10.0.0.0/1665 536 IPsVPC principal (le plus courant)
10.0.1.0/24256 IPsSubnet (256 - 5 reservees = 251 utilisables)
10.0.2.0/24256 IPsAutre subnet

Regle : Le /16 c'est le VPC (large), les /24 c'est les subnets (plus petits). Plus le nombre apres / est grand, moins il y a d'IPs.

AWS reserve 5 IPs par subnet : la premiere (reseau), la derniere (broadcast), et 3 pour AWS (DNS, routeur, futur usage).

Subnets en detail

Un subnet = un sous-reseau dans UNE seule AZ. Un subnet ne peut PAS s'etendre sur plusieurs AZ. Mais tu peux avoir plusieurs subnets dans la meme AZ.
Subnet PUBLICSubnet PRIVE
Route tableRoute 0.0.0.0/0 → igw-xxxRoute 0.0.0.0/0 → nat-xxx (ou aucune)
IP publique ?Oui (auto-assign public IP active)Non (IP privee uniquement)
Accessible depuis Internet ?Oui (si SG le permet)Non
Peut sortir vers Internet ?Oui (via IGW)Oui seulement via NAT Gateway
On y met quoi ?ALB, Bastion Host, NAT GatewayEC2 appli, RDS, ElastiCache

Internet Gateway (IGW)

  • C'est la porte d'entree/sortie vers Internet pour ton VPC
  • Il y a 1 IGW par VPC (pas plus)
  • Il est hautement disponible par defaut (AWS le gere)
  • Pour qu'un subnet soit public, sa route table doit avoir : 0.0.0.0/0 → igw-xxx

NAT Gateway

  • Permet aux instances dans un subnet prive d'acceder a Internet (ex: pour telecharger des mises a jour)
  • SANS que Internet puisse les contacter (sens unique : sortie seulement)
  • Se place dans un subnet public (il a besoin de l'IGW)
  • Coute de l'argent (~$0.045/h + cout du trafic)
  • Pour la haute dispo : un NAT Gateway par AZ
Schema logique du flux reseau

Instance privee → Route Table (0.0.0.0/0 → NAT) → NAT Gateway (dans subnet public) → Route Table (0.0.0.0/0 → IGW) → Internet Gateway → Internet

Route Tables (tables de routage)

  • Chaque subnet est associe a exactement 1 route table
  • Une route table contient des regles de routage (destination → cible)
  • Route table publique : 10.0.0.0/16 → local + 0.0.0.0/0 → igw-xxx
  • Route table privee : 10.0.0.0/16 → local + 0.0.0.0/0 → nat-xxx
  • La route local permet la communication entre tous les subnets du VPC

Elastic IP (EIP)

  • Adresse IP publique statique (ne change pas au reboot)
  • Tu peux l'attacher a une EC2 ou un NAT Gateway
  • Gratuite quand elle est utilisee, payante si elle ne l'est pas (AWS ne veut pas de gaspillage d'IPs)

Security Groups (SG) — EN DETAIL

Security Group = firewall VIRTUEL au niveau de l'instance. Chaque instance EC2 a au moins 1 SG.
ProprieteDetail
NiveauInstance (ENI = Elastic Network Interface)
StatefulSi tu autorises le trafic ENTRANT, la reponse sort automatiquement (pas besoin de regle sortante)
ReglesUniquement des regles ALLOW (pas de DENY possible)

VPC Peering

  • Connexion reseau directe entre 2 VPC (meme compte ou differents, meme region ou differentes)
  • Non transitif : si VPC A ↔ VPC B et VPC B ↔ VPC C, alors A ne peut PAS parler a C (il faut un peering A ↔ C)
  • Les CIDR des 2 VPC ne doivent pas se chevaucher
QCM : Un serveur dans un subnet prive doit telecharger des mises a jour depuis Internet. Quel composant est necessaire ?
R : NAT Gateway (place dans un subnet public)
QCM : Tu veux que ta BDD MySQL n'accepte des connexions que depuis ton serveur web EC2. Comment configurer le SG de la BDD ?
R : Inbound rule : TCP port 3306, source = le Security Group du serveur web (sg-xxx)
QCM : Quelle est la difference fondamentale entre Stateful et Stateless ?
R : Stateful (SG) = la reponse est automatiquement autorisee. Stateless (NACL) = tu dois autoriser entree ET sortie explicitement
QCM : Un VPC A est peere avec B, et B est peere avec C. A peut-il communiquer avec C ?
R : Non. Le VPC peering est non transitif. Il faut un peering direct A ↔ C

C'est quoi ?

EC2 = machine virtuelle (serveur) que tu loues a la demande. Tu choisis l'OS, la taille, le stockage, et le reseau.

S3 Static Website Hosting

  • S3 peut heberger un site web statique (HTML, CSS, JS) sans serveur
  • URL format : http://bucket-name.s3-website-region.amazonaws.com
  • Il faut activer le static website hosting ET rendre les objets publics (bucket policy)

S3 Bucket Policy vs ACL

  • ACL = permissions plus simples et plus anciennes. AWS recommande de les desactiver
  • Block Public Access = parametres au niveau du compte/bucket pour empecher tout acces public accidentel

Amazon Aurora

  • Version AWS de MySQL/PostgreSQL, 5x plus rapide que MySQL, 3x que PostgreSQL
  • Stockage auto-scale de 10 Go a 128 To
  • 6 copies des donnees sur 3 AZ automatiquement
  • Jusqu'a 15 Read Replicas (vs 5 pour RDS classique)
  • Aurora Serverless : scaling automatique, paie a l'usage

Amazon DynamoDB

  • BDD NoSQL (cle-valeur et document) geree par AWS
  • Serverless (pas de serveur a gerer), scaling automatique
  • Latence en millisecondes a n'importe quelle echelle
  • Modele : Table → Items (lignes) → Attributes (colonnes)
  • Cle primaire obligatoire : Partition Key (+ optionnel : Sort Key)
  • Capacite : On-Demand (auto) ou Provisioned (tu fixes les Read/Write Units)
  • DAX = cache en memoire pour DynamoDB (latence microseconde)

R : Creer des Read Replicas et diriger les lectures vers elles
QCM : Quel service de BDD pour des donnees cle-valeur avec une latence < 10ms et un scaling automatique ?

7. IAM — Identity and Access Management — EN DETAIL

Les 4 composants

ComposantDefinitionDetail
UserUne personne ou applicationA des credentials (password pour console, access key pour CLI/API). 1 user = 1 personne
GroupCollection d'utilisateursLes policies attachees au groupe s'appliquent a tous ses membres. Ex: groupe "Developers"
RoleIdentite temporairePour les services AWS (EC2, Lambda) ou les utilisateurs federes. Pas de credentials permanentes
PolicyDocument JSON de permissionsDefinit Effect (Allow/Deny) + Action (s3:GetObject) + Resource (arn:aws:s3:::mon-bucket/*)

Structure d'une policy IAM

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::mon-bucket/*"
  }]
}

Cette policy autorise la lecture de tous les objets dans "mon-bucket".

Types de policies

TypeGere parExemple
AWS ManagedAWS (precreee)AmazonS3ReadOnlyAccess
Customer ManagedToiPolicy custom pour ton application
InlineToi, attachee a 1 seul user/group/rolePermission tres specifique pour 1 ressource

Principes de securite IAM

  1. Principe du moindre privilege — Donner uniquement les permissions strictement necessaires
  2. Ne JAMAIS utiliser le compte root pour les taches quotidiennes — Creer un IAM admin a la place
  3. Activer MFA (Multi-Factor Authentication) sur root et les comptes sensibles
  4. Utiliser des Roles pour les services — EC2 qui accede a S3 → IAM Role, pas des access keys dans le code
  5. Rotation des cles — Changer regulierement les access keys
  6. Audit avec CloudTrail — Chaque appel API est logue

Role vs User — Quand utiliser quoi ?

SituationUtiliserPourquoi
Personne qui se connecte a la consoleIAM UserCredentials permanentes (password + MFA)
EC2 qui doit acceder a S3IAM RoleCredentials temporaires, pas de cles dans le code
Lambda qui ecrit dans DynamoDBIAM RoleService AWS → toujours un role
Application externe qui accede a AWSIAM Role + STSFederation, credentials temporaires

Modele de responsabilite partagee

AWS est responsable de la securite DU cloud :
  • Hardware physique, datacenters, reseau global
  • Virtualisation, hyperviseur
  • Regions, AZ, Edge Locations
TOI tu es responsable de la securite DANS le cloud :
  • Configuration des SG, NACL
  • Gestion IAM (users, roles, policies)
  • Chiffrement des donnees
  • Patching de l'OS (si EC2 = IaaS)
  • Configuration du firewall (SG, NACL)
  • Formation des utilisateurs
QCM : Une instance EC2 a besoin de lire des objets dans un bucket S3. Quelle est la methode la plus securisee ?
R : Creer un IAM Role avec la policy s3:GetObject et l'attacher a l'instance EC2
QCM : Un developpeur a accidentellement rendu public un bucket S3. Qui est responsable ?
R : Le client (toi). La configuration des ressources est ta responsabilite (modele de responsabilite partagee)
QCM : Quel outil AWS permet de tracer chaque appel API effectue sur le compte ?
R : AWS CloudTrail

8. Load Balancing + Auto Scaling — EN DETAIL

Elastic Load Balancer (ELB)

Un load balancer repartit le trafic entre plusieurs instances pour eviter la surcharge d'un seul serveur.

Types de Load Balancer

TypeCouche OSIProtocolesFonctionnalitesCas d'usage
ALBCouche 7HTTP, HTTPS, WebSocketRoutage par URL, host, header. Target groupsSites web, API REST, microservices
NLBCouche 4TCP, UDP, TLSUltra-rapide (millions req/s), IP statiqueGaming, IoT, performance extreme
GLBCouche 3IPRoutage vers appliances virtuellesFirewalls, IDS/IPS

Concepts ALB

  • Listener : regle qui ecoute sur un port (ex: port 80 → redirect vers HTTPS, port 443 → forward vers target group)
  • Target Group : groupe d'instances qui recevront le trafic. Avec health checks
  • Health Check : l'ALB ping regulierement chaque instance (ex: GET /health). Si elle ne repond pas, l'ALB arrete de lui envoyer du trafic
  • Cross-zone load balancing : repartit le trafic uniformement entre toutes les AZ
  • Sticky Sessions : un utilisateur est toujours envoye sur la meme instance (cookie)

L'ASG ajoute ou retire automatiquement des instances EC2 selon la demande.

ALB + ASG ensemble

Utilisateurs → ALB → Target Group → ASG (min=2, max=10, desired=2)
Quand le trafic augmente : ASG lance de nouvelles instances → les enregistre dans le target group → ALB repartit le trafic dessus. Quand le trafic baisse : ASG termine les instances superflues.

En QCM, la bonne reponse est presque toujours le scaling HORIZONTAL (ASG).

QCM : Comment gerer automatiquement les pics de trafic pour un site web ?
R : ALB + Auto Scaling Group avec une target tracking policy (ex: CPU a 50%)
QCM : Le health check de l'ALB echoue pour une instance. Que se passe-t-il ?
R : L'ALB retire l'instance du target group (plus de trafic envoye). L'ASG peut la terminer et en lancer une nouvelle
QCM : Quelle est la difference entre un ALB et un NLB ?
R : ALB = couche 7 (HTTP/HTTPS, routage par URL). NLB = couche 4 (TCP/UDP, ultra-performant)

10. Mise en cache — CloudFront et ElastiCache

Amazon CloudFront (CDN)

CloudFront = Content Delivery Network. Des copies de ton contenu dans 400+ points de presence dans le monde.

Comment ca marche

  1. L'utilisateur demande une page web
  2. CloudFront cherche dans l'edge location la plus proche
  3. Si le contenu est en cache → le retourne immediatement (cache hit)
  4. Si non en cache → va le chercher a l'origin (S3, EC2, ALB) → le met en cache → le retourne

Concepts cles

ConceptDefinition
DistributionLa configuration CloudFront (origin, comportement du cache, TTL)
OriginLa source du contenu (S3 bucket, EC2, ALB, HTTP externe)
Edge LocationPoint de presence ou le contenu est cache
TTLTime to Live : combien de temps le contenu reste en cache
InvalidationForcer la suppression du cache (payant)
OAC/OAIPermettre a CloudFront d'acceder a un bucket S3 prive

Cas d'usage typique : S3 (site statique) + CloudFront = site web rapide dans le monde entier avec HTTPS gratuit (certificat ACM).

Amazon ElastiCache

ElastiCache = cache en memoire RAM pour les requetes de BDD frequentes.

RedisMemcached
PersistenceOui (donnees sauvees sur disque)Non (tout en memoire)
ReplicationOui (Multi-AZ, Read Replicas)Non
Types de donneesStrings, listes, sets, hashesStrings uniquement
Cas d'usageSessions, classements, queuesCache simple
QCM : Comment reduire la latence pour des utilisateurs en Asie accedant a un site heberge en Europe ?
R : Amazon CloudFront (cache le contenu dans les edge locations proches de l'Asie)
QCM : La BDD est surchargee par des requetes repetitives. Comment reduire la charge ?
R : Ajouter Amazon ElastiCache (Redis ou Memcached) devant la BDD

11. Architectures decouplees — SQS, SNS, Lambda

Amazon SQS (Simple Queue Service)

  • File d'attente de messages geree par AWS
  • Producteur envoie un message → SQS le stocke → Consommateur le lit et le supprime
  • Standard Queue : debit illimite, au moins une livraison, ordre non garanti
  • FIFO Queue : ordre garanti, exactement une livraison, 300 msg/s max
  • Visibility Timeout : temps pendant lequel un message lu est invisible aux autres consommateurs
  • Dead Letter Queue (DLQ) : queue ou vont les messages qui echouent apres N tentatives
  • Retention : 1 minute a 14 jours (defaut : 4 jours)

Amazon SNS (Simple Notification Service)

  • Pub/Sub : 1 message → N abonnes
  • Un topic recoit des messages et les distribue a tous les abonnes
  • Abonnes possibles : SQS, Lambda, email, SMS, HTTP endpoint
  • Cas d'usage : notifications, alertes, fan-out (1 event → plusieurs actions)

Pattern SNS + SQS (fan-out)

Evenement → SNS Topic → SQS Queue 1 (traitement commande)
                      → SQS Queue 2 (envoi email)
                      → SQS Queue 3 (analytics)

Un seul message publie sur SNS est distribue a 3 queues SQS independantes.

AWS Lambda

  • Executer du code sans serveur (serverless). Tu upload ton code, AWS l'execute
  • Pay per invocation : tu paies uniquement quand le code tourne (nombre de requetes + duree)
  • Langages : Python, Node.js, Java, Go, .NET, Ruby
  • Duree max : 15 minutes par execution
  • Memoire : 128 Mo a 10 Go
  • Triggers : API Gateway, S3, SQS, SNS, CloudWatch Events, DynamoDB Streams...
QCM : Comment decoupler un service de traitement de commandes d'un service d'envoi d'email ?
R : Mettre une queue SQS entre les deux. Le service commande envoie un message, le service email le consomme
QCM : Quel service pour executer du code en reponse a un upload S3, sans gerer de serveur ?
R : AWS Lambda (triggered par un evenement S3)

13. Securite avancee

AWS Secrets Manager

  • Stocker des secrets (mots de passe BDD, cles API) de maniere securisee
  • Rotation automatique des secrets (ex: changer le mot de passe RDS tous les 30 jours)
  • Acces via API : GetSecretValue (l'app demande le secret au runtime)
  • C'est ce qu'on utilise dans le mini-projet pour le mot de passe de la BDD RDS

AWS KMS (Key Management Service)

  • Gerer les cles de chiffrement
  • Utilise par S3, EBS, RDS, etc. pour le chiffrement au repos
  • CMK (Customer Master Key) : ta propre cle ou une cle geree par AWS

AWS WAF (Web Application Firewall)

  • Protege les applications web contre les attaques (SQL injection, XSS, bots)
  • Se place devant CloudFront, ALB, ou API Gateway
  • Regles basees sur IP, geolocalisation, taille de requete, patterns

AWS Shield

  • Shield Standard (gratuit) : protection DDoS basique, automatique sur tous les services
  • Shield Advanced (payant, $3000/mois) : protection DDoS avancee, equipe de reponse, remboursement des couts lies a l'attaque

TypeQuandExemples

  • Detection de menaces intelligente
  • Analyse CloudTrail, VPC Flow Logs, DNS Logs avec du machine learning
  • Detecte : crypto mining, acces depuis des IPs malveillantes, comportements anormaux

AWS Inspector

  • Analyse automatique de vulnerabilites dans tes EC2 et conteneurs
  • Verifie les CVE, la configuration reseau, les bonnes pratiques

14. Couts et facturation — EN DETAIL

Modele de tarification AWS

  • Pay-as-you-go : payer ce qu'on consomme (pas d'engagement par defaut)
  • Payer moins en reservant : RI, Savings Plans
  • Payer moins a grande echelle : reductions de volume (S3 par exemple)
  • Trafic entrant : gratuit (data IN)
  • Trafic sortant : payant (data OUT vers Internet)
  • Trafic entre AZ : payant (petite charge)
  • Trafic dans la meme AZ : gratuit (si IP privee)

Outils de gestion des couts

Strategies de reduction des couts

  1. Reserved Instances / Savings Plans pour les charges stables (jusqu'a -72%)
  2. Spot Instances pour les charges interruptibles (jusqu'a -90%)
  3. Rightsizing : ne pas surdimensionner (utiliser CloudWatch pour analyser l'usage reel)
  4. S3 Lifecycle : deplacer les anciens fichiers vers Glacier
  5. Eteindre les ressources non utilisees (dev/test la nuit et le week-end)
  6. Utiliser les services manages (RDS vs MySQL sur EC2 : moins de temps d'admin)

AWS Organizations

  • Gerer plusieurs comptes AWS de maniere centralisee
  • Consolidated Billing : une seule facture pour tous les comptes (= reductions de volume)
  • Service Control Policies (SCP) : restreindre les services/actions autorisees par compte

17. MEGA Regles d'or pour les QCM

  1. Haute disponibilite → Multi-AZ (RDS), ALB + ASG (EC2), S3 (11 neufs)
  2. BDD en securite → Subnet prive + SG qui n'accepte que le SG du web server
  3. Credentials dans le code → JAMAIS. Utiliser Secrets Manager ou IAM Role
  4. Site statique → S3 + CloudFront
  5. Reduire les couts stables → Reserved Instances / Savings Plans
  6. Reduire les couts non critiques → Spot Instances

  9. Latence mondiale → CloudFront (CDN)
  10. DNS → Route 53
  11. Serverless → Lambda + DynamoDB + S3 + API Gateway
  12. Scaling → HORIZONTAL (ASG). Presque toujours la bonne reponse
  13. Multi-AZ RDS = haute dispo | Read Replica = performance lecture
  14. SG = stateful (instance) | NACL = stateless (subnet)
  15. EBS = 1 instance 1 AZ | EFS = N instances multi-AZ | S3 = objets HTTP
  16. Qui a fait quoi ? → CloudTrail
  17. Metriques et alarmes ? → CloudWatch
  18. Config conforme ? → AWS Config
  19. Protection DDoS → AWS Shield (Standard = gratuit)

  21. EC2 sans SSH → Systems Manager Session Manager
  22. Conteneurs sans serveur → Fargate
  23. Trafic entrant AWS → GRATUIT
  24. Responsabilite partagee → AWS = securite DU cloud / Toi = securite DANS le cloud
  25. Si "la reponse la plus economique" → S3, Lambda, Spot, ou Glacier selon le contexte
